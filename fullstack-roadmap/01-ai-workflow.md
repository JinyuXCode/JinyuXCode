# AI 协作学习工作流

## 你和 AI 的分工

| 工作 | 你负责 | AI 可以辅助 |
| --- | --- | --- |
| 需求拆解 | 必须先写初版 | 审查遗漏、补充边界 |
| 架构方案 | 判断是否适合项目 | 给方案、列取舍 |
| 数据建模 | 定义实体和关系 | 生成 SQL、指出索引 |
| API 设计 | 确定资源、权限、错误码 | 生成接口文档 |
| 代码实现 | 审查和整合 | 生成初版代码 |
| 测试 | 确定核心场景 | 生成测试样例 |
| 调试 | 运行、定位、验证 | 分析日志、给修复建议 |
| 部署 | 理解配置和风险 | 生成 Docker / Nginx 草稿 |

重点是：AI 负责提速，你负责判断。

## 每个功能的固定流程

```text
1. 写需求拆解
2. 让 AI 审查需求
3. 修改需求
4. 让 AI 给技术方案
5. 审查方案
6. 让 AI 写代码
7. 本地运行
8. 写测试或接口用例
9. 让 AI 做 code review
10. 修改并复盘
```

## 让 AI 审查需求的提示词

```text
我正在做一个前端转 Go 全栈练习项目。
技术栈：Go + Gin + PostgreSQL + Redis + React/Vue。

下面是我自己拆解的需求，请你先审查我的拆解是否完整。
不要直接写代码。

请按以下格式输出：
1. 需求遗漏
2. 数据库设计建议
3. API 设计建议
4. 权限和安全风险
5. 事务和并发风险
6. 建议的开发步骤
```

## 让 AI 生成方案的提示词

```text
基于已经确认的需求，请给出后端实现方案。

要求：
1. 使用 Go + Gin。
2. 使用 handler / service / repository 分层。
3. 数据库使用 PostgreSQL 或 MySQL。
4. 所有数据库操作接收 context.Context。
5. 明确错误码、状态码、权限规则。
6. 先输出目录结构、数据表、API、关键流程，不要直接写代码。
```

## 让 AI 生成代码的提示词

```text
请基于上面的方案生成代码。

要求：
1. 遵循 handler / service / repository 分层。
2. handler 只处理 HTTP 参数和响应。
3. service 处理业务规则。
4. repository 处理数据库访问。
5. 不要硬编码密钥和数据库配置。
6. 使用统一错误响应。
7. 给核心 service 生成单元测试。
8. 解释每个文件的职责。
```

## 让 AI 做 code review 的提示词

```text
请从后端工程质量角度 review 下面代码。

重点检查：
1. 安全问题
2. 权限绕过
3. SQL / ORM 使用问题
4. 事务边界
5. 并发问题
6. 错误处理
7. 日志和可观测性
8. 测试遗漏

请只输出问题、影响和修改建议。
```

## 你必须亲自判断的内容

- 业务实体是否拆对。
- 表关系是否合理。
- 权限是否覆盖所有接口。
- 错误码是否方便前端处理。
- 事务边界是否正确。
- AI 有没有过度设计。
- AI 有没有把安全配置硬编码。
- 测试是否真的覆盖核心路径。

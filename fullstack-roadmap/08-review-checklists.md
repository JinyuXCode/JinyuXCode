# 审查清单与验收标准

## 需求审查清单

- [ ] 角色是否明确。
- [ ] 前端入口是否明确。
- [ ] 核心实体是否明确。
- [ ] 字段是否够用。
- [ ] 状态流转是否明确。
- [ ] 接口列表是否完整。
- [ ] 权限规则是否覆盖读和写。
- [ ] 异常场景是否列出。
- [ ] 是否需要事务。
- [ ] 是否需要缓存。
- [ ] 验收标准是否可测试。

## API 审查清单

- [ ] URL 是否表达资源。
- [ ] HTTP method 是否合理。
- [ ] 状态码是否合理。
- [ ] 请求参数是否明确。
- [ ] 响应结构是否统一。
- [ ] 错误码是否稳定。
- [ ] 分页、排序、筛选是否统一。
- [ ] 认证方式是否明确。
- [ ] 权限失败返回是否明确。
- [ ] API 文档是否能给前端直接使用。

## 数据库审查清单

- [ ] 每张表是否有主键。
- [ ] 唯一字段是否有唯一索引。
- [ ] 高频查询字段是否考虑索引。
- [ ] 外键关系是否清楚。
- [ ] 多对多关系是否有中间表。
- [ ] 金额是否避免使用浮点数。
- [ ] 时间字段是否统一。
- [ ] 软删除策略是否明确。
- [ ] 数据库迁移是否可重复执行。
- [ ] 关键约束是否不只依赖代码判断。

## Go 代码审查清单

- [ ] handler 是否只处理 HTTP。
- [ ] service 是否只处理业务。
- [ ] repository 是否只处理数据访问。
- [ ] 是否正确传递 `context.Context`。
- [ ] 错误是否被包装或转换。
- [ ] 是否存在硬编码配置。
- [ ] 日志是否包含定位信息。
- [ ] 是否泄漏敏感信息。
- [ ] 是否有测试覆盖核心逻辑。
- [ ] package 是否过度拆分。

## 认证鉴权审查清单

- [ ] 密码是否哈希存储。
- [ ] token secret 是否来自配置。
- [ ] token 是否设置过期时间。
- [ ] token 中是否避免敏感信息。
- [ ] 未登录接口是否正确拦截。
- [ ] 无权限接口是否正确拦截。
- [ ] 权限是否在后端校验。
- [ ] 普通用户是否能越权访问他人数据。
- [ ] 管理员接口是否单独保护。
- [ ] 错误提示是否避免账号枚举。

## 事务和并发审查清单

- [ ] 多表写入是否需要事务。
- [ ] 状态变更是否校验当前状态。
- [ ] 库存或余额是否考虑并发。
- [ ] 支付或回调是否幂等。
- [ ] 重复提交是否处理。
- [ ] 失败后是否能回滚。
- [ ] 是否有唯一约束防止重复数据。
- [ ] 是否需要乐观锁或数据库锁。

## 缓存审查清单

- [ ] 这个接口是否真的需要缓存。
- [ ] cache key 是否稳定。
- [ ] TTL 是否合理。
- [ ] 数据更新后是否删除缓存。
- [ ] 是否可能缓存空值。
- [ ] Redis 异常是否降级。
- [ ] 是否把 Redis 当成主数据库。
- [ ] 是否存在缓存穿透、击穿、雪崩风险。

## 文件上传审查清单

- [ ] 文件大小是否限制。
- [ ] 文件类型是否限制。
- [ ] 文件名是否重新生成。
- [ ] 是否校验真实 MIME。
- [ ] 是否保存元信息。
- [ ] 是否有访问权限。
- [ ] 删除策略是否明确。
- [ ] 是否避免把可执行文件暴露给用户。

## 测试验收清单

- [ ] 正常路径测试。
- [ ] 参数错误测试。
- [ ] 未登录测试。
- [ ] 无权限测试。
- [ ] 数据不存在测试。
- [ ] 重复数据测试。
- [ ] 状态非法流转测试。
- [ ] 事务失败测试。
- [ ] 缓存失效测试。
- [ ] 部署启动测试。

## 作品集验收标准

一个合格的全栈作品至少包含：

- [ ] 可运行代码。
- [ ] README。
- [ ] `.env.example`。
- [ ] 数据库迁移文件。
- [ ] API 文档。
- [ ] 表结构说明。
- [ ] 权限说明。
- [ ] 测试说明。
- [ ] Docker 启动方式。
- [ ] 复盘文档。

优秀作品还应该包含：

- [ ] 状态流转图。
- [ ] 权限矩阵。
- [ ] 事务说明。
- [ ] 缓存策略。
- [ ] 部署地址。
- [ ] 截图或演示录屏。
- [ ] AI 协作记录和你的审查修改说明。

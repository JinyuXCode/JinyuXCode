# 项目 4：任务协作系统或电商后台

## 项目定位

这是第二个作品集项目，用来证明你不只是会 CRUD，而是能处理真实业务复杂度。

二选一即可，不建议两个都做。

## 选择 A：任务协作系统

适合训练 SaaS、后台系统、多角色协作。

### 核心实体

```text
users
workspaces
workspace_members
projects
tasks
task_comments
task_attachments
operation_logs
```

### 核心功能

- [ ] 创建工作空间。
- [ ] 邀请成员。
- [ ] 创建项目。
- [ ] 创建任务。
- [ ] 分配任务负责人。
- [ ] 任务状态流转。
- [ ] 评论任务。
- [ ] 上传附件。
- [ ] 操作日志。
- [ ] 工作空间权限。

### 状态流转

```text
todo -> in_progress -> review -> done
in_progress -> blocked
blocked -> in_progress
review -> in_progress
```

### 权限矩阵示例

| 动作 | Owner | Admin | Member | Guest |
| --- | --- | --- | --- | --- |
| 删除工作空间 | 是 | 否 | 否 | 否 |
| 邀请成员 | 是 | 是 | 否 | 否 |
| 创建任务 | 是 | 是 | 是 | 否 |
| 修改任务状态 | 是 | 是 | 是 | 否 |
| 查看任务 | 是 | 是 | 是 | 是 |

### 必须审查

- [ ] 用户是否只能访问自己所属工作空间。
- [ ] 成员权限是否由后端校验。
- [ ] 任务状态是否可以乱跳。
- [ ] 删除项目是否影响任务和附件。
- [ ] 操作日志是否记录关键动作。

### 最小验收

- [ ] 一个用户可以创建工作空间。
- [ ] 可以邀请另一个用户。
- [ ] 可以创建项目和任务。
- [ ] 可以推进任务状态。
- [ ] 非成员不能访问工作空间数据。

## 选择 B：轻量电商后台

适合训练交易、库存、订单、事务。

### 核心实体

```text
users
products
skus
inventory_records
orders
order_items
payments
operation_logs
```

### 核心功能

- [ ] 商品管理。
- [ ] SKU 管理。
- [ ] 库存管理。
- [ ] 创建订单。
- [ ] 取消订单。
- [ ] 模拟支付。
- [ ] 订单状态流转。
- [ ] 销售统计。
- [ ] 后台权限。

### 订单状态流转

```text
pending_payment -> paid -> shipped -> completed
pending_payment -> canceled
paid -> refunded
```

### 事务场景

创建订单时通常需要：

- [ ] 检查商品和 SKU 是否存在。
- [ ] 检查库存是否足够。
- [ ] 锁定或扣减库存。
- [ ] 创建订单。
- [ ] 创建订单明细。
- [ ] 写库存流水。

这些步骤需要重点考虑事务和并发。

### 必须审查

- [ ] 库存是否可能被超卖。
- [ ] 订单状态是否可以非法跳转。
- [ ] 支付回调是否幂等。
- [ ] 价格是否以后端为准。
- [ ] 普通用户是否能修改商品。

### 最小验收

- [ ] 可以创建商品和 SKU。
- [ ] 可以下单。
- [ ] 下单会扣减库存。
- [ ] 库存不足不能下单。
- [ ] 可以模拟支付。
- [ ] 订单状态流转受限制。

## AI 协作方式

先让 AI 审查你的拆解，不要直接写代码：

```text
下面是我的业务系统需求拆解。
请重点审查：
1. 实体关系是否遗漏
2. 状态流转是否合法
3. 哪些操作需要事务
4. 哪些接口存在越权风险
5. 哪些地方可能并发冲突
6. 最小 MVP 应该先做哪些功能
```
